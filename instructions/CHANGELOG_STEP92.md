
## Step 92 — Digital-file readback + publish-state readback (real root cause found via live Etsy API calls)
**Date:** 2026-07-08

---

### Context

Maj reported a real, live listing (task `fb66a81a`, "Customizable Family
Recipe Greeting Card") with 2 real listing photos but no downloadable
digital file. The assumed cause was a missing file-attachment readback
(the extended gate from step 91 verifies listing images and PDF page
counts, but never independently confirmed the digital file itself).

**That assumption turned out to be wrong.** Investigation via `railway ssh`
plus real, live calls to Etsy's own API (not assumption, not memory — the
project has been burned by assumed field/endpoint shapes multiple times
already) found a different, more interesting root cause.

---

### Step 1 — what actually happened (real evidence, in order)

1. **`ImageCatalogService.get_delivery_asset(task_id)`**: real, non-null
   asset (`design.png`, generated by `PODDesignAgent`). A deliverable file
   WAS generated.
2. **Was the digital-file upload actually called?** Yes — production log
   table shows `PipelineOrchestrator: ... stages_ok: ['listing_images',
   'delivery_asset', 'create_listing', 'attach_publish']` for this task,
   and the deployment's raw HTTP logs show
   `POST .../shops/58716525/listings/4534427807/files "HTTP/1.1 201 Created"`.
3. **Did the upload response indicate real success?** A live, direct call
   to Etsy's `getAllListingFiles` endpoint (`GET
   /v3/application/shops/{shop_id}/listings/{listing_id}/files` — verified
   against Etsy's real published OpenAPI spec, which also confirms this
   operation IS shop-scoped, unlike the images equivalent) returned:
   ```json
   {"count":1,"results":[{"listing_file_id":1499536729759,"listing_id":4534427807,
     "filename":"design.png","filesize":"537.46 KB","size_bytes":550361, ...}]}
   ```
   **The file was genuinely attached from the very first pipeline run.**
   Ticket's hypothesis (file never uploaded) is disproven by Etsy's own
   ground truth.
4. **The real bug**: a live `GET /v3/application/listings/4534427807`
   returned `"state": "edit"` — the listing was still a **draft**, never
   actually live/purchasable, despite `PipelineOrchestrator`'s own report
   claiming `attach_publish` succeeded. `EtsyImageService.publish_listing()`'s
   PATCH call to activate the listing had returned HTTP 200 — but the
   **response body's own `state` field**, which the code never checked,
   would have shown the transition didn't take effect. Re-invoking the
   identical PATCH later (during this investigation) returned
   `"state": "active"` and the listing became genuinely live at
   `https://www.etsy.com/listing/4534427807/...` — proving the underlying
   listing data was always fine; the activation simply didn't take effect
   at the moment of the original attempt, most likely a brief
   eventual-consistency lag immediately following the image/file uploads
   that happen in the same call sequence right before it.

**Root cause: `publish_listing()` trusted the HTTP status code and never
validated the response body's actual `state` — the same class of "blind
trust in an apparent success signal" bug as the earlier
missing-`shop_id`-in-`publish_listing()` bug, just a different failure
mode (a silent no-op instead of a wrong endpoint).**

**Important disclosure:** the diagnostic PATCH call made during this
investigation (needed to inspect the real response body) had the side
effect of actually completing the listing's activation — it is now
genuinely live on Etsy, with real photos and a real file, but that
happened as an unintended byproduct of debugging, not as a verified
outcome of a passing gate. Maj should treat this the same as the prior
bad-listing cleanup: still worth taking down/reviewing manually, since it
was never validated end-to-end by a gate that has since been fixed.

---

### Fix

**`app/services/etsy_image_service.py`**:
- `publish_listing()` now checks the response body's real `state` field
  (not just HTTP status) and **retries once after a 2-second delay**
  before giving up — absorbing the observed propagation lag. Returns an
  accurate `published: bool` reflecting ground truth.
- New `get_listing_files(listing_id)` — readback verification for the
  digital file, mirroring the existing `get_listing_images()` pattern.
  Verified shop-scoped per Etsy's real OpenAPI spec (`getAllListingFiles`),
  unlike `get_listing_images`/`getListingImages` which is not — checked
  directly rather than assumed, after getting burned by an assumed
  endpoint shape before.

**`app/services/pipeline_orchestrator.py`** (`_stage_attach_publish`):
- New file-readback check: after a successful (non-error) digital upload,
  calls `get_listing_files()` and requires at least one file present.
  Failure → same "worse than no listing" handling as every other readback
  gate: delete the listing, mark `BLOCKED_NO_PRODUCT`.
- New publish-state check: if a real publish was attempted
  (`AUTO_PUBLISH_LISTINGS=True`) and `publish_result["published"]` is not
  `True`, same failure handling. This is the check that would have caught
  the actual production incident.

Both checks slot into the existing sequence (generate → validate →
register → upload/attach → **independently confirm via Etsy's own
response** → only then consider the stage complete) established in step 91.

### Tests

`scripts/test_step92_file_and_publish_readback.py` (new, 4/4 pass):
1. `publish_listing()`: PATCH returns `state: "edit"` on attempt 1,
   `"active"` on retry → succeeds, `published: True` (proves the actual
   fix for the production scenario).
2. `publish_listing()`: stuck on `"edit"` through both attempts →
   `published: False` (proves it doesn't blindly trust a 200 status).
3. Orchestrator: digital-file upload call "succeeds" (no error), but
   `get_listing_files()` readback reports zero files → listing deleted,
   task blocked (the ticket's explicit ask).
4. Orchestrator: `publish_result` reports `published: False` → listing
   deleted, task blocked (reproduces the actual production bug end-to-end
   through the orchestrator, with the fix in place).

All 17 existing test suites re-run and still pass (several test doubles'
fixed `publish_result` dicts needed `published: True` added, since the
previous shape — a raw Etsy response with no `published` key at all when
`AUTO_PUBLISH_LISTINGS=True` — always evaluated to `False` and was itself
part of why this bug went unnoticed: the orchestrator's own report field
`"published": result.get("publish_result", {}).get("published", False)`
was ALWAYS false on a real publish attempt even before this fix, since the
raw Etsy listing object was never guaranteed to carry a `published` key).

═══════════════════════════════════════════════════════════════

AI Factory is a solo-developer automation platform for a single Etsy
shop (shop_id 58716525) whose END STATE is a fully autonomous business
that requires NO human-created product content. Concretely:

  - The app finds/decides what to sell (market intelligence agents,
    already partially built — full autonomous triggering is step 88,
    not part of this session).
  - The app CREATES THE ACTUAL PRODUCT ITSELF. Two product types, both
    in scope long-term:
      (a) DIGITAL DOWNLOADS — planners, printables, templates,
          checklists, wall art. The deliverable file itself (a PDF or
          high-res print-ready image) is generated by AI and is what
          the customer receives. This is the priority for THIS
          session (steps 67-76) because it requires no external
          fulfillment partner and can be finished end-to-end now.
      (b) PRINT-ON-DEMAND PHYSICAL GOODS — mugs, shirts, etc. The
          product is an AI-generated design that gets sent to a POD
          service for printing and shipping. The design-generation
          side is in scope now (step 71); the actual POD
          service integration (submitting orders, tracking
          fulfillment) is step 81, LATER, not in this session. Do not
          build a fake/partial POD fulfillment integration to
          compensate — just make sure the design artifact this
          session produces is stored in a form step 81 can consume
          later (see step 71/75 notes below).
  - The app writes SEO copy and creates the Etsy listing (already
    built, steps 55-58) — but currently only as a DRAFT. Making a
    listing actually go live/sellable is explicitly part of THIS
    session's scope (folded into step 73 — see below) because it
    requires the image pipeline this session builds.
  - The app markets it (Pinterest, already built, steps 59-61).
  - The app tracks what's working (analytics/revenue/performance,
    already built, steps 62-65).

The person running this (Maj) does not write, design, or upload any

═══════════════════════════════════════════════════════════════
2. HARD SYSTEM RULES (from markdown.txt — do not violate these)
═══════════════════════════════════════════════════════════════

- No direct agent-to-agent communication — everything goes through
  the orchestrator / task pipeline.
- Agents never select a model or provider directly — that's the
  provider manager's job (ProviderManager for text,
  ImageProviderManager for images, added step 66).
- Agents request capabilities (generate_text, reason, generate_image,
  vision, embed, classify), never provider names.
- All structured outputs must be strict JSON matching a schema — no
  markdown, no explanations, no extra text around it.
- Every output must pass schema validation. Failure path is:
  retry → repair → retry → fail. Never silently accept malformed output.

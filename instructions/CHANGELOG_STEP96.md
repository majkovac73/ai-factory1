
## Step 96 — Real content-quality gate + marketing/deliverable consistency
**Date:** 2026-07-08

---

### Step 1 — root cause of the "2 þutter" text corruption

Traced through the code (mechanism unambiguous):

- The delivery asset for `greeting_card_design` (every single-image format)
  is generated by `PODDesignAgent` → the **image-generation model**
  (Seedream 4.5) from a prompt containing only `product_name` + a generic
  `visual_brief`. For task fb66a81a the description was *"…personalize it
  with your favorite family recipe…"* — **no actual recipe text anywhere**.
  The image model invented and rendered the recipe text itself, and
  image-generation models cannot reliably render text — hence "2 þutter"
  (thorn glyph for 'b'), a duplicated ingredient, a trailing dash.
- **Not** a PIL/font bug: only `pdf_generation_service.py` draws text via
  PIL; the single-image delivery path never does.
- **Not** a content-templating bug: no raw recipe text exists in the
  codebase — it exists only as pixels in the generated PNG.

**Root cause: reliance on an image model to render text-heavy content, with
no check that reads the resulting pixels.** Every prior gate is structural.
Fix = a content-inspection gate (Step 2), not a generation/font change.

### Step 2 — the content-quality gate (core of this fix)

`app/services/content_quality_service.py` (new) — first check that inspects
actual content. Sends the asset to a **vision-capable model** via new
`OpenRouterProvider.generate_with_images()`, returns strict JSON:
`text_legible` / `text_coherent` / `matches_intended_content` /
`specific_issues`; `passed` False if any boolean fails. Prompt explicitly
hunts image-model text artifacts (garbled/corrupted glyphs, duplicated
lines, cut-off text). Images downscaled to 1024px to keep per-call cost
predictable.

Wired into `PipelineOrchestrator` as a **mandatory stage (2.5)** — after
delivery-asset generation, **before any Etsy/Printify upload**, earlier than
every structural gate. On failure: regenerate + re-review up to
`CONTENT_QA_MAX_ATTEMPTS` (default 2), then block. PDFs are QA'd **per page**
inside `PDFGenerationService` before assembly (garbled page → regenerate →
fail whole PDF).

New settings: `CONTENT_QA_MODEL="openai/gpt-4o-mini"`,
`CONTENT_QA_MAX_ATTEMPTS=2`.

### Step 3 — marketing/deliverable mismatch (chosen: (a) + (b))

With a text-to-image-only stack (no img2img), option (b) alone is weak
(retrying decoupled marketing generation won't make it match). So:

- **(a) structural:** for digital single-image products, the QA-passed
  **delivery asset is prepended as the primary listing photo** — the
  featured photo is now, by construction, exactly what the buyer receives.
  (Prepended for upload order only; NOT re-registered in the catalog, which
  would clobber the `delivery` record the hard gate needs.)
- **(b) backstop:** `check_marketing_consistency()` vision-compares the
  listing photos vs the delivery asset and blocks on clearly unrelated
  content (allowing presentation differences), retry-then-block.

### Cost note

New mandatory per-product cost, `openai/gpt-4o-mini` + 1024px downscale
(estimated from gpt-4o-mini published pricing and its ~33.3× image-token
multiplier — **confirm on first real run**):

- ~**$0.004 per image review** (1024px ≈ 25.5K input tokens).
- **Single-image digital:** content review + consistency ≈ **+$0.012–0.016**;
  prior ~$0.16 → **~$0.175** (≈ +10%).
- **PDF (≤6 pages):** per-page reviews + consistency ≈ **+$0.03**; prior
  ~$0.36 → **~$0.39**.

Well within `MAX_DAILY_SPEND_USD=5.00`. Tunable via `CONTENT_QA_MODEL`; the
consistency call could be dropped (delivery-as-primary-photo already
guarantees the featured photo matches) if cost needs cutting.
